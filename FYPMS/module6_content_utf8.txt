Smart FYP Management System FYPMS — Refined Functional Requirements Module 6: Academic Batch & Curriculum Lifecycle Management Version 2.0  •  Refined & Dependency-Resolved Edition  1. Executive Analysis — Current Problem Statement  After thorough analysis of the existing requirements (Modules 1–5 completed, Modules 6–14 pending), the following critical gaps and dependencies were identified that block system-wide functionality: Core Problem Identified Module 6 in its original form assumes that batches, academic years, and FYP-I/FYP-II transitions are already set up — but no module currently handles the CREATION and MANAGEMENT of these academic structures. Without a proper Batch & Academic Session management layer, Modules 7 through 11 (submissions, evaluations, transitions) cannot function correctly. 1.1 Identified Gaps in Existing Modules The following gaps exist in Modules 1–5 that need MINIMAL additions to support Module 6 dependencies. These are additive-only — no existing logic is changed: Module Gap / Missing Minimal Addition Required Module 1 — Authentication No concept of which batch/session a student belongs to at login Student account must carry batch_id & fyp_phase (FYP-I or FYP-II) as profile fields Module 2 — Profile Management Student profile has no academic session reference Add batch enrollment linkage to student profile (read-only; set by Admin) Module 4 — Proposal Management Proposals have no batch/session association Proposals must be tagged with active batch_id so they are scoped correctly Module 5 — Supervision Supervisor workload is not scoped per academic session Workload counter must be session-scoped (reset per batch, not globally) 2. Module 6 — Refined: Academic Batch & Curriculum Lifecycle Management  Module Scope (Refined) This module is now re-scoped as the FOUNDATION module for academic operations. It manages: (1) Academic Batch creation and lifecycle, (2) FYP-I and FYP-II phase assignment, (3) Department-specific milestone tracks and weekly task configuration, (4) Smart deadline management, and (5) New Batch onboarding with FYP-I to FYP-II transition authority. Admin and Coordinator both have defined, non-overlapping authorities. 2.1 Academic Batch Creation & Lifecycle Management The system shall allow the Admin to manage academic batches (cohorts of students) with full lifecycle control. A Batch is the top-level organizational unit that drives all downstream functionality. Req. ID Requirement Description Depends On Priority FR6.1.1 The system shall allow Admin to create a new Academic Batch with a unique Batch Name (e.g., 'BSCS-2025-Batch'), associated department(s), academic year, and start date. Each batch must be assigned a phase: FYP-I or FYP-II. Module 1, Module 2 HIGH FR6.1.2 The system shall allow Admin to enroll students into a batch by bulk CSV upload (using SAP ID as the primary key). The system must validate each SAP ID against existing user accounts and reject unknown SAP IDs with per-row error reporting. FR6.1.1, Module 1 HIGH FR6.1.3 The system shall allow Admin to set a batch to one of the following lifecycle states: Draft, Active, Frozen, or Archived. Only one batch per department per phase may be in Active state at a time. FR6.1.1 HIGH FR6.1.4 The system shall prevent any student submission, evaluation, or milestone activity if the batch is in Draft or Archived state, returning a clear error message to the student. FR6.1.3, Module 7 HIGH FR6.1.5 The system shall allow Admin to view a Batch Overview Dashboard showing: batch name, phase, state, enrolled student count, milestone completion percentage, and active/overdue task count. FR6.1.1 MEDIUM FR6.1.6 The system shall allow Admin to edit batch metadata (name, academic year, associated departments) only while the batch is in Draft state. Editing Active or Frozen batches shall be restricted to authorized Admin accounts with audit logging. FR6.1.3 MEDIUM FR6.1.7 The system shall maintain a complete audit log of all batch state transitions (who changed it, when, and from which state to which state). Module 13 MEDIUM 2.2 New Batch Onboarding — When a New Cohort Arrives When a new batch of students arrives each academic year, the Admin must perform a structured onboarding sequence. This subsection defines that workflow precisely. Req. ID Requirement Description Depends On Priority FR6.2.1 The system shall allow Admin to initiate a 'New Batch Onboarding' wizard that guides through: (a) creating the batch record, (b) enrolling students via CSV, (c) assigning a milestone track, and (d) activating the batch in a single structured flow. FR6.1.1, FR6.1.2 HIGH FR6.2.2 Upon batch activation, the system shall automatically assign the department-specific milestone track (16-week template) to all enrolled students in that batch, based on their department field in their SAP ID profile. FR6.1.3, FR6.3.1 HIGH FR6.2.3 The system shall automatically set all newly enrolled students in an active FYP-I batch to fyp_phase = 'FYP-I' in their profile, making this field visible to supervisors and coordinators but read-only for students. FR6.1.2, Module 2 HIGH FR6.2.4 The system shall send automated email notifications to all students enrolled in a newly activated batch, informing them of their batch name, FYP phase, assigned department track, and first active task. FR6.2.2, Module 12 HIGH FR6.2.5 The system shall allow Admin to view a pre-activation checklist before activating a batch, showing: students enrolled count, milestone track assigned (yes/no), supervisor capacity status, and any configuration warnings. FR6.1.1, Module 5 MEDIUM 2.3 FYP-I to FYP-II Transition Management This is one of the most critical dependency chains in the system. The transition from FYP-I to FYP-II must be structured, auditable, and must carry forward all relevant data from the completed phase. Transition Authority — Who Can Do What ADMIN: Can create the FYP-II batch, approve/deny transition requests, enroll students, and set batch as Active. COORDINATOR: Can flag groups as 'Eligible for Transition', raise transition issues, and reopen deadline extensions for transition review. STUDENT: Can only VIEW their transition status. They cannot self-initiate or self-approve transition. Req. ID Requirement Description Depends On Priority FR6.3.1 The system shall allow Admin to mark an existing FYP-I batch as 'Eligible for Transition' once all required milestone submissions are complete (auto-check: zero unsubmitted mandatory tasks). A manual override with justification is allowed if auto-check fails. FR6.1.3, Module 7 HIGH FR6.3.2 The system shall allow Admin to create a corresponding FYP-II batch linked to the originating FYP-I batch. The FYP-II batch must inherit: (a) all enrolled student SAP IDs, (b) approved project records (title, group composition), and (c) assigned supervisor assignments. FR6.3.1, Module 11 HIGH FR6.3.3 The system shall automatically carry forward the fyp_phase field for each student from 'FYP-I' to 'FYP-II' upon Admin confirmation of the transition, without requiring manual per-student updates. FR6.3.2, Module 2 HIGH FR6.3.4 The system shall prevent a student's FYP-II batch from being activated if their FYP-I proposal was never approved (status != Approved). Admin must explicitly override with documented justification before activation proceeds. FR6.3.2, Module 4 HIGH FR6.3.5 The system shall allow Admin to handle transition exceptions: students who failed FYP-I may be (a) held back in a repeat FYP-I batch, (b) excluded from the FYP-II batch entirely, or (c) given a conditional carry-forward with supervisor acknowledgment. Each exception must be logged. FR6.3.2 HIGH FR6.3.6 The system shall allow Coordinator to raise a 'Transition Issue Flag' against a specific group (e.g., incomplete hardcopy forms, attendance shortfall) that blocks that group's carry-forward until the Admin resolves it. FR6.3.2, Module 9 MEDIUM FR6.3.7 Upon successful FYP-II batch activation, the system shall send automated transition confirmation emails to all transitioned students, their supervisors, and the coordinator, listing the carried-forward project title and group members. FR6.3.3, Module 12 HIGH FR6.3.8 The system shall maintain a complete Transition Audit Record per batch, showing: source FYP-I batch, target FYP-II batch, transition date, Admin who approved, total students transitioned, exception count, and override justifications. Module 13 MEDIUM 2.4 Department-Specific Milestone Track Configuration The Coordinator configures the 16-week milestone tracks per department per phase. These tracks are the blueprints that drive all student submission and supervisor evaluation timelines. Req. ID Requirement Description Depends On Priority FR6.4.1 The system shall allow Coordinator to create a Milestone Track Template for a specific department and FYP phase (FYP-I or FYP-II), consisting of up to 16 weekly task slots. Each track must have a unique name and be linked to a department. FR6.1.1 HIGH FR6.4.2 The system shall allow Coordinator to define each weekly task within a track with the following attributes: (a) week number (1–16), (b) task title, (c) task type (Template-Based / Instruction-Only / Attendance), (d) mandatory or optional flag, (e) release rule (auto or manual), (f) deadline offset from batch start date (in days). FR6.4.1 HIGH FR6.4.3 The system shall allow Coordinator to upload downloadable task templates (PDF or DOCX, max 10 MB) for Template-Based tasks, with version control. Each uploaded template must be versioned, timestamped, and the previous version archived — not deleted. FR6.4.2 HIGH FR6.4.4 The system shall allow Coordinator to define Instruction-Only tasks that provide written instructions without requiring a downloadable template, but still accept student document uploads. FR6.4.2 MEDIUM FR6.4.5 The system shall allow Coordinator to clone an existing Milestone Track Template and modify it for a new batch or department, rather than building from scratch. The cloned track must be independent of the source. FR6.4.1 MEDIUM FR6.4.6 The system shall prevent deletion of a Milestone Track Template that is currently assigned to an Active batch. The Coordinator must first unassign the track before deletion is permitted. FR6.4.1, FR6.1.3 HIGH FR6.4.7 The system shall allow Admin to assign a specific Milestone Track Template to a batch during onboarding or after batch creation (but before activation). Once a batch is Active, the track assignment may only be changed by Admin with audit logging. FR6.4.1, FR6.2.1 HIGH 2.5 Smart Deadline Management & Task Release Rules All deadlines are computed dynamically from the batch start date and configured offsets. The system enforces deadlines automatically with no manual intervention required for normal operation. Req. ID Requirement Description Depends On Priority FR6.5.1 The system shall automatically compute and set each task's release date and deadline date based on the batch start date plus the configured offset (in days) defined in the Milestone Track Template. FR6.4.2, FR6.1.3 HIGH FR6.5.2 The system shall automatically release tasks to students on their computed release date without requiring manual Coordinator intervention, and send a release notification email to all affected students and their supervisors. FR6.5.1, Module 12 HIGH FR6.5.3 The system shall display a Smart Deadline Timer (countdown) to students for each active task, turning yellow when 48 hours remain and red when 24 hours remain before the deadline. FR6.5.1, Module 7 MEDIUM FR6.5.4 The system shall automatically lock a task for submission once its deadline passes, record the lock event with a timestamp in the audit log, and apply a 'Late Submission' flag to any submission received after the deadline but before a reopening. FR6.5.1, Module 13 HIGH FR6.5.5 The system shall allow Coordinator to reopen a deadline for a specific student group (identified by SAP ID or group ID) with a mandatory written justification, new deadline datetime, and automatic logging of: who approved, when, and for which group. FR6.5.4 HIGH FR6.5.6 The system shall allow Coordinator to manually trigger task release for tasks configured with a manual release rule, preventing them from auto-releasing until the Coordinator explicitly publishes the task. FR6.4.2, FR6.5.1 MEDIUM FR6.5.7 The system shall allow Admin to perform a global deadline extension for an entire batch (e.g., due to university holiday or force majeure), which shifts all remaining task deadlines by a specified number of days, with full audit logging. FR6.5.1, Module 13 HIGH 2.6 Master Completion Dashboard & Compliance Monitoring The Coordinator needs real-time visibility into all groups' progress across all departments. This subsection defines the reporting and monitoring requirements. Req. ID Requirement Description Depends On Priority FR6.6.1 The system shall provide a Master Completion Dashboard for Coordinators showing, per active batch: (a) total groups, (b) groups with all mandatory tasks submitted, (c) groups with one or more overdue mandatory tasks, (d) overall completion percentage. FR6.1.5, Module 7 HIGH FR6.6.2 The system shall allow Coordinator to filter the dashboard by department, week number, task type, and submission status (Submitted / Pending / Late / Missing) using SAP ID as an additional search filter for individual student lookup. FR6.6.1 HIGH FR6.6.3 The system shall allow Coordinator to export a Compliance Report (CSV) per batch showing: Group ID, SAP IDs of all members, task name, submission status, and submission date for all mandatory tasks. FR6.6.1, Module 5 MEDIUM FR6.6.4 The system shall automatically flag a group as a 'Compliance Gap' if one or more mandatory tasks are overdue by more than 3 days, and surface these flags prominently on the Coordinator dashboard with email alert to the Coordinator. FR6.6.1, Module 12 HIGH FR6.6.5 The system shall allow Admin to access a cross-batch view showing active batches across all departments side-by-side with their completion status, for executive oversight. End-to-End System Operation — How It All Works  The following describes the complete lifecycle flow of the Smart FYPMS with Module 6 as the foundation layer, from new batch creation to FYP-II completion: Phase A — New Academic Year Begins (Admin Actions) Admin creates new Academic Batch record: e.g., 'BSCS-2026-FYP-I', assigns department(s), sets start date. Admin bulk-enrolls students via CSV upload (SAP IDs validated against Module 1 user accounts). Admin assigns the department-specific Milestone Track Template (configured by Coordinator) to the batch. Admin reviews pre-activation checklist: students enrolled, track assigned, supervisor capacity adequate. Admin activates batch. System auto-sets all enrolled students to fyp_phase = 'FYP-I'. System sends batch activation emails to all students with their track, phase, and first task details. Phase B — Active FYP-I Semester (Coordinator & Supervisor Actions) Tasks auto-release each week per the configured schedule. Coordinator manually triggers any manual-release tasks. Students submit weekly milestones via Module 7. System enforces deadlines — submissions lock after deadline. Supervisors review and evaluate submissions via Module 8. Feedback is linked to group SAP IDs. Coordinator monitors Master Completion Dashboard for compliance gaps and overdue submissions. Coordinator reopens deadlines for specific groups with justified exceptions if needed. Admin can issue global deadline extensions if university-wide disruptions occur. Phase C — FYP-I End & Transition to FYP-II (Admin Actions) Admin checks batch completion: system auto-verifies that all mandatory tasks are submitted. Coordinator raises Transition Issue Flags for groups with pending hardcopy forms or attendance shortfalls. Admin resolves flags, handles exceptions (hold-backs, conditional carry-forwards), and logs justifications. Admin creates corresponding FYP-II batch, system auto-carries forward: students, supervisors, approved proposals. Admin activates FYP-II batch. System updates all student profiles to fyp_phase = 'FYP-II'. Supervisor workload counters reset for the new FYP-II batch. FYP-II milestone track is assigned. Transition confirmation emails sent to all students, supervisors, and coordinator. Phase D — FYP-II Completion & Archival FYP-II follows same weekly milestone cycle. Module 10 handles rubric-based final evaluation. Upon completion, Admin sets batch state to 'Archived'. Module 14 archives all projects with full metadata. Projects become searchable via Module 3 Project Archive for future students and faculty.
